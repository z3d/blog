Consistency checks

Looking at what code actually does

A practical way to spot drift in agent-written code before it becomes the new normal.

The problem

Generated code can be locally fine and collectively tiring

Each file can compile, pass tests, and answer the ticket. The pain appears later, when neighbouring files no longer agree on the shape of the work.

One handler validates first

Another loads first. Both might be defensible. Together, they make the next change slower.

One query caches a list

The sibling query doesn't. Nobody wrote the rule down, so the difference slips through.

One config hides a class

Every other EF configuration has one file per class. The exception sits there quietly.

Plain claim

Think of it as a nudge in review

Pick a family of similar files, pin a few examples of the intended shape, inspect the rest, and tell reviewers which files look unlike their peers.

A score gets you in the neighbourhood. The useful part is the explanation: seven dependencies, no entity loads, and the only try/catch in the cohort.
Cohorts first

Compare like with like

The measurement only makes sense inside a cohort: command handlers with command handlers, query handlers with query handlers, EF configurations with EF configurations.

34 command handlers in the case-study cohort
20 query handlers, split across by-id, list, and paginated-list shapes
26 EF configurations, measured by fluent API shape
Reference point

Pin exemplars before measuring the rest

Current average

The average can already contain drift. Once a bad pattern is in the reference, the next bad pattern starts to look ordinary.

or

Pinned exemplars

A small set of files the team would defend as the intended shape, with written justification for each one.

Structural fingerprint

A fingerprint is a row of boring facts

The fingerprint is an inventory. It extracts small, concrete facts that describe the file's shape.

How it runs

The pipeline is deliberately mechanical

Find peers Discover the files in one cohort.
Pin examples Use the written exemplar list as the reference.
Inspect code Reflect over compiled types and source metadata.
Normalise Put counts and booleans on comparable scales.
Explain Show the features that drove the difference.
Important lesson

Normalise first or the biggest file wins

The first command-handler run put byte-size next to booleans. Unsurprisingly, IlByteSize explained almost everything.

Before

IlByteSize contributed 98.5% of the distance score. The report had quietly become a large-file sorter.

After

Feature contributions spread across entity-load count, byte size, try/catch, and dependency count. The result became a shape report.

Supporting layers

Shingles and source tokens are supporting signals

The structural fingerprint did most of the useful work. The other layers helped at the margins, and their limits matter.

IL shingles

Compare short opcode sequences. They are a cheap second opinion for compiled skeleton shape, and often move with the structural score.

Source-token similarity

Compare local vocabulary from names, strings, type references, and metadata. Useful for hunches. Weak evidence for intent.

Tests as governance

The tests protect the measuring instrument

These tests keep the report honest and stop it drifting into policy by accident. Unusual files still go to a person.

Exemplar governance

  • Every exemplar resolves to a real file.
  • Every exemplar has a written justification.
  • The centroid stays anchored to exemplars.

Advisory boundary

  • Hard "distance < 5.0" merge gates stay out.
  • Known recurring findings graduate to convention tests.
  • Distance remains a prompt for human review.
From signal to rule

The best findings stop being scores

When the same signal fires for a clear reason, the right move is to make a deterministic convention test and retire the guesswork.

Inline DTO construction

Found through review of source-token outliers, then enforced as CommandHandlers_MustNotConstructMappedDtosInline.

List-query caching

Started as an advisory concern. Became a convention test once the invalidation risk was clear.

One EF config per file

A nested configuration was surfaced by exemplar discovery and split into its own file.

Keep it modest

An outlier deserves a look

The report can flag legitimate complexity. The two SERIALIZABLE transaction handlers looked unusual because they were unusual. That helped us understand them, with no rewrite implied.

The failure mode to avoid

A distance gate punishes the first good example of a new pattern and rewards the tenth copy of a bad one. Keep this in review output and away from merge blocking.

Where it fits

Put it between review instinct and hard convention tests

Problem Best tool Why
Rule already known Convention test Deterministic yes or no
File feels unlike its peers Consistency report Ranks attention and explains differences
Business case is genuinely odd Human review note The code may be right because the domain is odd
Practical verdict

Keep the boring parts. Question the fancy parts.

Use this

  • Cohort-specific discovery.
  • Pinned exemplars with reasons.
  • Simple structural fingerprints.
  • Per-feature divergence reports.
  • Advisory output.

Treat carefully

  • Composite scores without explanations.
  • Small exemplar sets pretending to be robust models.
  • Vocabulary similarity dressed up as intent.
  • Build failures based on distance alone.
Takeaway

The idea is useful because it is humble

Look at the code. Count the small facts. Compare it with chosen examples. Show reviewers where the shape has changed.

The score gets attention. The conversation does the work.
